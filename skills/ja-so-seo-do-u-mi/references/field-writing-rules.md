# Field Writing Rules

Practical heuristics for turning a weak 자기소개서 draft into a clearer, evidence-led application essay. This file is a runtime rule reference for the skill, not a bibliography. Do not browse external videos or websites while using these rules unless the user explicitly asks for fresh research.

## Rules

### F-1. Start With What The Reader Needs

The first sentence should reveal the answer, relevant strength, or core evidence. Avoid opening with long background.

Weak:

```text
대학교 3학년 때 팀 프로젝트를 진행한 경험이 있습니다.
```

Better:

```text
반복되는 일정 지연을 역할별 체크리스트로 줄이며 프로젝트를 마감한 경험이 있습니다.
```

### F-2. Result And Action Beat Intention

Hiring readers cannot evaluate "열심히", "관심", or "성실" unless they see action and result.

Rewrite abstract intent into:

```text
problem -> user's action -> observed change
```

For future-oriented prompts, use the same principle:

```text
role expectation -> first action -> expected work output
```

Do not stop at `개선하겠습니다`, `기여하겠습니다`, or `성장하겠습니다`.

### F-3. Use Numbers Only When Provided

Numbers improve credibility, but invented numbers destroy trust. If the draft has no numbers, use qualitative evidence:

- 마감 전 제출
- 반복 오류 감소
- 고객 문의 유형 정리
- 팀 내 기준 합의
- 담당자 간 역할 충돌 해소

### F-4. Show "I" Inside Team Experience

Team stories must identify the user's actual contribution:

- 내가 정리한 것
- 내가 제안한 것
- 내가 맡은 범위
- 내가 바꾼 방식
- 내가 확인한 결과

### F-5. Use Why-Drilling

When a draft is vague, infer the strongest cause chain from the text:

```text
왜 이 경험이 중요했나?
왜 이 행동을 했나?
왜 이 결과가 직무와 연결되나?
```

Do not invent answers. If the chain is missing, ask for the missing evidence or keep the claim modest.

### F-6. Subtitle Only When It Helps Scanning

Subtitles can help, but slogan-like subtitles often sound artificial. Prefer concise evidence subtitles:

- `반복 지연을 줄인 일정 관리`
- `고객 문의를 유형화한 분석 경험`

Avoid:

- `성장을 향한 끝없는 도전`
- `소통으로 만든 아름다운 결과`

### F-7. Final Essay Should Feed The Interview

Every strong claim should be defensible in an interview. Flag claims that invite questions the user cannot answer from the provided draft.

### F-8. Keep Concrete Detail Role-Aware

Concrete detail is not the same in every job. Do not force one job's detail style onto another role unless the JD and draft make it relevant.

Choose details from the role's natural work:

- who the work was for
- what problem repeated
- what the applicant changed or decided
- what artifact or output remained
- what constraint mattered
- what changed afterward
